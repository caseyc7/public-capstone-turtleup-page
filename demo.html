Development Process

Our application development follows an agile methodology with three planned sprints, with the first sprint being devoted to planning and very early initial development work, and the latter 2 being dedicated to full-on implementation

Sprint 1: Initial Planning and Use Case Requirements

Initial Goal from Client

During our initial sprint, our client outlined their initial goals and plans for the project.

  • Integrate the app with the tracker being built by a separate engineering team
  • Allow users to view the location data for specific turtles in real time
  • Integrate existing features from the TurtleUp webpage into the app (Shop, About/Info Page)

Additionally, the client had requested an iOS and Android compatible version of the app. The original plan for this was to create 2 seperate teams, each creating their own version of the app. While this was intriguing from a learning perspective, as it would allow each team to devote their full attention to each aspect of the app in a full-stack capacity, it became clear this was unfeasable for multiple reasons.

Working as 2 seperate entities would not only result in increased costs for the client (depending on the different design decisions that were made between teams), but would more than likely result in entirely different user experiences across platforms. To remedy this, we combined into a larger overall group, while still delegating individual tasks among our smaller groups. This allows us to work independently while also making sure that everyone understands the use cases and requirements correctly. We believe that this process will allow to create a more desirable product for our client and the product's eventual users.

Technical Planning

As we decided to build a cross platform app, we decided on using React-Native to best fit our needs. Other options included Swift (iOS), Flutter, or Kotlin, but those options were either not suitable for our needs, or not well supported.

Using React-Native presented some challenges at first, but we were able to adapt. Some of these include:

  • Understanding the basic structure of react based apps
  • Knowing the strengths and limitations of certain features across platforms
  • Understanding how to integrate with our planned backend implementation

On the backend side of development, we opted to use Firebase from Google Cloud as our Database. The main advantage of this was that Firebase was well-suited for real-time updates, which our app was going to require as one of its main features.

Along with this, we planned to write our backend in Golang, focusing on future stability and speed, ensuring the app remains quick and easy to develop once we are no longer working on it.

Our Contributions

During Sprint 2, we were able to implement all of the features listed above as a larger group. As a smaller group, however, we were tasked with building the home page, sign in page, and parts of the info/about page.

Home Page

The main goal with the home page was to display all of the turtles we stored in our DB in an intuitive manner. The way we decided to design this was to format each turtle document from Firebase into its own "card" that contained all/most of the information contained in the schema.

We also wanted to make the page somewhat interactive, so we decided on adding a "My Turtles" and "Explore Turtles" tab at the top of the page. The Explore tab would display all turtles to any user, and the My Turtles tab would display only turtles that are connected to the user. While it has not yet been decided how turtles will be connected to users from a business perspective, our next steps for this page will be actually filtering turtles based on the currently logged in user.

Lastly, icons were added that represent a photo and a map. The photo icon loads an image (specific to the turtle you are currently selecting) from a Firebase storage bucket, and displays it to the user. The next step for this function will be to cache the photos on device to prevent too many reads to the bucket itself. The map icon currently redirects the user to the map page. The next steps for this feature will be to display the selected turtle's location data on screen after being redirected.

Current state of the Home Page after Sprint 2

Sign Up Page

The main challenge with the home page was understanding home Firebase handles User Auth. Becuase the service already has its own built in way to handle authentication, we decided to use this feature for out own app. Once we understood how to use the packages Firebase provides for working with react-native, most of the work came down to styling and making the page look visually appealing

There was some communication that had to occur between both sides of the larger group. Most of this happened due to resolving issues connecting the signup and login process, as well as understanding how to use oAuth for allow for signing in with Google, which ultimately presented some issues with cost and cross-compatability.

Sign Up Page (Newsletter Implementation)

The signup page was mostly implemented last sprint, with the only feature missing was connecting the newsletter signup checkbox to an actual api endpoint. This sprint, we spent time talking with our client to learn about their current newsletter signup implementation and how it works.

After learning that TurtleUp is using mailchimp, we began implementing an API route that would allow us to connect to TurtleUp's already existing mailchip service. This way, a user could opt to sign up for the newsletter directly in the app. This posed challenges in learning how to set up an API to an external service other than our DB, which had already been set up. However, we were able to perform testing with a seperate Mailchimp account and eventually connect that to the real mailchimp service once that was fully working. We also added fields to the sign-up page for first and last name to allow for personalized newsletters.

Additional Contributions

Along with the above features, we completed some smaller code re-factors meant to enhance user experience and future expansion of the app. One of these was allowing the products stored in our shop page to be pulled directly from our DB. Previously, we had products statically stored in the assets folder of our app. Now, we have a schema in firebase that links to photos stored in a google cloud storage bucket. This will allow for future products to be easily added and removed.

When this was first implemented, there were issues with requests being sent to firebase for a new collection of images even when all of those images had been read and downloaded already, resulting in redundant requests that could eventually lead to excess costs and storage issues. Now, the images are grabbed when the page is initally loaded and only re-downloaded in the user leaves and re-enters the page.

Overall Project Takeaways

Overall

We felt that working with TurtleUp was an incredibly rewarding experience. Being given the opportunity to build an app of this magnitude was a significant challenge that we believed prepared us for future projects and challenges we may encounter

Future Considerations

There were some things we could have done differently, but we largely believe that the way we tackled the project was well thought out considering our experience with these technologies and platforms at the time of the project beginning. One thing in particular that we could have tackled differently was completing documentation along the way. Specifically, creating documentation for new features as we created them. We always knew this was going to be a project that would be passed on to future capstone groups, so documentation has always been a necessity, and creating the documentation early on would have caused less headaches at the end of the project while attempting to rememeber everything we did.

Closing Thoughts

As this app gets passed down to future groups, it will be interesting to see how they begin to tackle challenges like itegration with a real satellite tracker, which has always been the end goal for this project. While we did not get to take this on ourselves, we feel confident that we left the project in a good enough spot that future features can be (relatively) easily implemented.
